01

How reward changes everything

The post-training algorithms (PPO, GRPO) are the same ones from Lectures 3–4. What reasoning models changed is the reward: instead of a learned, subjective reward model, RLVR scores a completion with a deterministic verifier — extracted_answer == 77 or "all unit tests pass." A clean, ungameable signal is what opened the door to reasoning.

RLVR feedback loop: learned reward vs. verifiable reward

Same RL loop, one swapped block. A reward model emits a continuous score the policy can hack — drift toward high-scoring-but-wrong outputs (over-optimization). A verifier emits a binary 1/0 that only fires when the answer is actually correct. Flip the reward type and watch the training signal — and what the policy is incentivized to do — change on a toy math task.

In practice · Math and code are unusually lucky: a checkable verifier exists. Most real-world knowledge work — finance, healthcare, legal analysis — has no clean checkable answer, so you're stuck with a learned, gameable reward model (or a human) and have to invest accordingly in reward QA.

02

Reasoning: thinking tokens & where the loss lands

A reasoning model emits a long <think> chain-of-thought before the final answer. The crucial RLVR fact: the reward is computed only on the extracted outcome, yet the policy-gradient credit from that single scalar is spread across every token — including all the thinking tokens that were never directly scored.

What thinking tokens look like (and where the reward applies)

Compare a standard model (answers directly) with a reasoning model (thinks first). Step the reasoning trace token-by-token. The verifier reward fires only on the boxed outcome, but in policy-gradient RL that one reward weights the log-prob gradient of all tokens in the trace — so longer thinking is reinforced when it leads to a correct answer.

03

Inference-time (test-time) scaling

o1/R1 surfaced a second scaling axis. Train-time scaling bakes capability into the weights once. Test-time scaling spends more compute per query at fixed weights — a longer thinking budget. Accuracy climbs with the thinking budget, but with diminishing returns: the curve is concave and saturates at the model's competence ceiling.

Thinking-token budget → accuracy (diminishing returns)

At fixed weights, increase the chain-of-thought length budget and watch pass@1 accuracy rise then flatten. The model's capability ceiling (set by training) caps it — so two curves are shown: a weaker base model and a stronger reasoning-trained model. More thinking helps both, but cannot lift a model past its trained ceiling.

pass@1 vs pass@k: the coverage headroom

Test-time scaling also includes sampling many answers. pass@1 is single-shot accuracy; pass@k asks whether any of k samples is correct. The unbiased estimator from n samples with c correct is 1 − C(n−c, k)/C(n, k). The gap between pass@1 and pass@k is the headroom a selector or majority vote can chase.

05

One algorithm detail: CISPO vs ratio clipping

A recurring recipe choice from the deck: GRPO/PPO clips the per-token probability ratio, which can zero out the gradient on the very tokens that update the most. CISPO (MiniMax-M1) instead clips the importance-sampling weight while keeping every token's gradient alive. DAPO relaxes the upper clip. Drag the ratio to see which contributes a gradient.

Ratio clipping vs CISPO weight clipping

The probability ratio is r = πθ(a)/πold(a). PPO's surrogate min(r·A, clip(r,1−ε,1+ε)·A) kills the gradient outside the trust region (flat → zero slope). CISPO keeps the policy-gradient term A·∇logπ but only clips the IS weight as a multiplier — so high-update tokens still learn. Widen the upper clip to mimic DAPO's relaxed clipping.
